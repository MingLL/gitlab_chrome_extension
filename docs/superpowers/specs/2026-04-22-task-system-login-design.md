# 任务系统登录与未完成任务列表设计

## 目标

为当前 Chrome 插件增加“连接任务系统并拉取未完成任务”的能力。

用户在侧边栏中完成 GitLab 仓库和分支选择后，可以在同一界面内：

- 配置任务系统地址、账号和密码
- 自动获取验证码并完成登录
- 拉取当前用户的开发任务列表
- 仅查看未完成任务
- 手动选择一条任务，作为后续一键处理流程的输入

本次设计只解决“登录到获取任务列表”的闭环，不负责根据任务自动匹配分支，也不负责提交后续任务处理请求。

## 背景

当前插件已经具备以下能力：

- 连接 GitLab
- 选择仓库和分支
- 获取仓库 clone 地址与最新提交 hash
- 对发布平台页面执行一键填表

但用户后续操作仍依赖另一个内部任务系统。当前流程需要用户单独打开任务网站、登录、筛选自己的开发任务，再回到插件中继续执行。这个过程分散、重复，也容易在多个系统之间来回切换时出错。

任务系统接口链路已经明确：

1. `GET /session/verificationCode`
2. 从响应读取 `data.uuid` 与 `data.realcode`
3. `POST /session/login`
4. 使用 `code`、`codeUuid`、`loginName`、`loginPwd` 登录
5. 从响应读取 `data.xaccessToken`
6. `POST /task/page/querymydevtask`
7. 请求头兼容 `X-Access-Token` 与 `Cookie: token=...`

其中验证码接口返回 `realcode`，因此当前场景可以自动完成验证码识别，无需人工输入。

## 范围

本次包含：

- 在侧边栏中新增任务系统配置与任务列表区域
- 保存任务系统地址、账号、密码等配置
- 自动请求验证码并使用 `realcode` 登录
- 登录成功后查询“我的开发任务”
- 仅展示未完成任务
- 支持用户手动选择任务
- 为登录、查询、空结果和失败状态提供明确提示
- 补充 API client、过滤逻辑与界面交互测试

本次不包含：

- 根据任务字段自动推导 GitLab 仓库
- 根据任务字段自动推导分支
- 自动选择任务
- 自动提交任务处理请求
- 修改任务系统中的任务状态
- 多任务批量处理

## 用户目标

用户希望：

- 不离开插件侧边栏就能拿到自己的未完成任务
- 不需要手工识别验证码
- 选中一个任务后，继续配合自己手动选择的仓库和分支执行后续流程
- 当任务系统登录失效或网络异常时，能看到清晰原因

## 方案比较

### 方案一：在现有侧边栏中新增任务系统区域

做法：

- 在现有 GitLab 选择流程下方追加任务系统配置、刷新按钮和任务列表
- 用户仍在一个页面内完成仓库、分支和任务选择

优点：

- 与当前操作链路最贴近
- 后续接入“一键处理任务”最顺畅
- 不需要新增页面路由或额外视图切换

缺点：

- 侧边栏状态会变多，需要仔细拆分状态边界

### 方案二：为任务系统增加独立视图

做法：

- 将任务系统做成单独页面或页签，与 GitLab 选择区分开

优点：

- 结构清晰，关注点分离明显

缺点：

- 交互链路被切断
- 用户在任务和 GitLab 之间来回切换的成本更高

### 方案三：只先做底层 client，不做完整任务列表 UI

做法：

- 先实现接口调用和存储，仅保留一个简单验证入口

优点：

- 实现快

缺点：

- 无法直接支持用户当前真正需要的“列出未完成任务并选择”

## 推荐方案

采用方案一：在现有侧边栏中新增任务系统区域。

原因：

- 用户当前的真实流程就是在同一插件里衔接仓库、分支和任务
- 现有侧边栏已经承载主要交互，新增区域的学习成本最低
- 后续任务处理按钮需要同时依赖“已选仓库、已选分支、已选任务”，统一放在一个界面里最容易协同

## 交互设计

## 任务系统配置区

在侧边栏中新增一块“任务系统”配置区，字段包括：

- `任务系统地址`
- `登录账号`
- `登录密码`

交互说明：

- 地址默认可预填为 `http://10.254.239.10:10086`
- 账号和密码支持本地保存，减少重复输入
- 点击 `刷新任务` 时自动执行验证码、登录、查任务整条链路

## 任务列表区

配置区下方展示任务状态与列表：

- 加载中提示
- 空列表提示
- 错误提示
- 未完成任务列表
- 当前选中任务摘要

列表首版展示字段：

- `proposalname`
- `taskjobid`
- `env`
- `system`
- `starttime`
- `endtime`
- `proposalstatusdetail`
- `completed`

用户点选某条任务后：

- 该任务成为当前选中任务
- 侧边栏中保留简要摘要
- 后续流程可以直接读取这条任务

## 使用流程

1. 用户连接 GitLab 并选择仓库、分支
2. 用户在任务系统区域确认地址、账号、密码
3. 点击 `刷新任务`
4. 插件自动获取验证码并登录
5. 插件查询“我的开发任务”
6. 插件过滤出未完成任务并展示
7. 用户手动选择一条任务
8. 后续动作从当前选中任务读取上下文

## 数据流设计

请求编排固定为以下顺序：

1. 请求验证码接口
2. 读取 `uuid` 与 `realcode`
3. 发起登录请求
4. 提取 `xaccessToken`
5. 组装兼容请求头
6. 查询开发任务列表
7. 过滤未完成任务
8. 更新侧边栏状态

任务查询默认使用当前已确认的请求体结构：

```json
{
  "startDeploytime": "",
  "endDeploytime": "",
  "endOnlinedate": "",
  "startOnlinedate": "",
  "actualName": "",
  "filterType": "0",
  "pageNum": 1,
  "pageSize": 18,
  "proposal": "",
  "searchCount": true,
  "taskjobid": "",
  "titletype": ""
}
```

说明：

- 首版先按现网页行为对齐，避免引入额外筛选差异
- 如果后续需要支持翻页、搜索或更多筛选，再在此基础上扩展

## 未完成任务过滤策略

当前已知任务字段中包含 `completed`，本次以该字段作为未完成判断的主依据。

默认策略：

- 将“已完成”状态排除
- 其余任务视为待处理并展示

实现上建议将过滤逻辑独立封装，避免把状态判断散落在 UI 中。这样如果后续发现还需要结合 `proposalstatusdetail` 或其他字段判断，只需调整单个函数。

## 模块设计

建议拆分为三个主要模块。

### 1. 任务系统 API client

职责：

- 获取验证码
- 发起登录
- 组装认证头
- 查询开发任务
- 规范化接口错误

建议位置：

- `src/lib/task-system/client.ts`
- `src/lib/task-system/types.ts`
- `src/lib/task-system/mappers.ts`
- `src/lib/task-system/errors.ts`

### 2. 任务系统配置存储

职责：

- 持久化任务系统地址、账号、密码
- 管理最近一次成功登录使用的配置

建议位置：

- `src/lib/storage/taskSystemConfigStorage.ts`

说明：

- 密码是否明文存储应在实现时明确权衡
- 若延续当前插件简单本地配置模式，可先与已有配置存储方式保持一致

### 3. 侧边栏任务区域

职责：

- 渲染配置表单
- 触发刷新任务
- 展示加载与错误状态
- 渲染未完成任务列表
- 管理当前选中任务

建议位置：

- `src/sidepanel/components/TaskSystemForm.tsx`
- `src/sidepanel/components/TaskList.tsx`
- `src/sidepanel/components/SelectedTaskSummary.tsx`
- `src/sidepanel/App.tsx`

## 类型设计

建议增加独立类型，减少 UI 对原始接口结构的直接依赖。

```ts
export type TaskSystemCredentials = {
  baseUrl: string;
  loginName: string;
  loginPwd: string;
};

export type TaskSystemSession = {
  token: string;
};

export type TaskSummary = {
  id: string;
  proposalId: string;
  proposalName: string;
  proposalType: string;
  proposalStatusDetail: string;
  system: string;
  env: string;
  taskJobId: string;
  testManager: string;
  startTime: string;
  endTime: string;
  completed: string | number | boolean | null;
};
```

说明：

- client 层负责把接口原始字段映射成前端稳定字段
- UI 层只消费 `TaskSummary`

## 错误处理

错误分为四类。

### 配置错误

例如：

- 任务系统地址为空
- 账号为空
- 密码为空

处理方式：

- 在提交前校验
- 阻止发起请求并展示就地提示

### 登录链路错误

例如：

- 验证码接口失败
- 登录接口失败
- 登录响应里缺少 token

处理方式：

- 统一在 client 层转为可读错误
- 在侧边栏展示“登录失败：xxx”

### 查询错误

例如：

- 任务接口超时
- 认证失效
- 服务端返回异常

处理方式：

- 首次查询失败时，可自动重新登录一次再重试一次
- 若仍失败，则将错误暴露给用户

### 数据错误

例如：

- 接口成功但 `list` 缺失
- 单条任务缺少关键展示字段

处理方式：

- 对分页结果做结构校验
- 对字段缺失使用合理降级展示

## 安全与兼容性

任务系统当前为 HTTP 明文地址，且登录密码为明文提交。

这意味着：

- 插件只应在明确受信任的内网环境中使用该功能
- 本次实现不额外改变现网安全模型，只复用网页现有调用方式

认证兼容策略：

- 任务接口请求头同时携带 `X-Access-Token`
- 若 token 可用，则同时拼接 `Cookie: token=<token>`

这样可以尽量贴近现有网页请求行为，降低接口兼容风险。
