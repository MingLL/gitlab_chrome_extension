# 仓库搜索与分支活跃排序设计

## 目标

优化 Chrome 插件中仓库和分支的选择体验，让用户在发布场景下更快找到目标仓库和当前活跃分支。

本次设计聚焦两个核心方向：

- 仓库支持搜索，并按照本地使用习惯进行个性化排序
- 分支支持搜索，并按照 GitLab 接口返回的最新提交时间倒序排列

## 背景

当前插件中的仓库和分支都使用原生下拉框选择。

现状问题：

- 仓库较多时，原生下拉不方便快速定位
- 分支较多时，缺少搜索与活跃度排序，发布时查找效率低
- 现有仓库排序仅区分“最近使用”和“全部项目”，个性化程度不够
- 分支如果按本地使用次数排序，会受到历史行为影响，不适合发布时寻找当前活跃分支

## 范围

本次包含：

- 将仓库选择改为“搜索框 + 结果列表”
- 将分支选择改为“搜索框 + 结果列表”
- 为仓库增加本地使用统计，并基于本地使用次数排序
- 为分支增加按最新提交时间倒序排序
- 保留仓库和分支的搜索过滤能力
- 更新界面文案与测试

本次不包含：

- 复杂模糊搜索算法
- 手动切换排序方式的控件
- 分支本地使用次数统计
- 基于 GitLab 提交作者数据的仓库热度统计
- 分页或虚拟滚动优化

## 用户目标

### 仓库选择场景

用户希望：

- 能直接输入仓库名或路径快速定位目标仓库
- 常用仓库排在前面，不必每次从头寻找
- 当前浏览器标签页匹配到的仓库可以优先出现

### 分支选择场景

用户希望：

- 在发布时快速找到最近仍在活跃的分支
- 通过分支名搜索快速缩小范围
- 列表默认就是“最近有提交的分支在前面”

## 推荐方案

采用“搜索框 + 结果列表”的面板式选择，而不是继续增强原生 `select`。

原因：

- 更适合承载搜索、排序、高亮和状态提示
- 更容易扩展后续键盘交互
- 结果列表可以展示更多辅助信息，例如路径、最新提交时间、短 hash
- 避免原生下拉在分组和自定义表现上的限制

## 交互设计

## 仓库区域

结构：

- 标题与说明文案
- 仓库搜索输入框
- 仓库结果列表
- 状态提示

搜索框建议文案：

- placeholder：`搜索仓库名称或路径`

列表项展示：

- 仓库名称
- `pathWithNamespace`
- 可选的辅助标签，例如 `常用` 或 `最近使用`

点击结果项后：

- 立即选中该仓库
- 加载该仓库的分支列表
- 更新仓库本地使用统计

## 分支区域

结构：

- 标题与说明文案
- 分支搜索输入框
- 分支结果列表
- 状态提示

搜索框建议文案：

- placeholder：`搜索分支名称`

列表项展示：

- 分支名称
- 最新提交时间
- 可选展示短 hash

点击结果项后：

- 立即选中该分支
- 更新结果汇总区的分支名与最新提交 hash

## 空状态与加载状态

仓库搜索无结果时显示：

- `没有匹配的仓库`

分支搜索无结果时显示：

- `没有匹配的分支`

保留现有加载状态提示，例如：

- `正在加载仓库...`
- `正在加载分支...`

## 排序规则

## 仓库排序

仓库排序基于本地行为记录进行个性化排序。

默认优先级：

1. 当前标签页匹配的仓库优先
2. `useCount` 倒序
3. `lastUsedAt` 倒序
4. 仓库名称字母序

说明：

- 仓库允许按本地使用习惯排序，因为常用仓库具有明显的个体偏好
- 这里的“使用”定义为：用户在插件中明确选中该仓库，并使其成为当前有效仓库

## 分支排序

分支排序不使用本地使用次数，而是使用 GitLab 接口返回的最新提交时间。

默认优先级：

1. 最新提交时间倒序
2. 分支名称字母序

说明：

- 分支的活跃程度具有明显时效性
- 某些历史常用分支在功能上线后会迅速失去参考价值
- 发布场景更适合优先展示最近仍有提交的分支

## 搜索规则

## 仓库搜索

匹配字段：

- 仓库名称
- `pathWithNamespace`

匹配方式：

- 大小写不敏感
- 包含匹配即可

示例：

- 输入 `infra` 可以匹配仓库名包含 `infra` 的项目
- 输入 `team/service-a` 可以匹配对应路径前缀

## 分支搜索

匹配字段：

- 分支名称

匹配方式：

- 大小写不敏感
- 包含匹配即可

示例：

- 输入 `release` 可以匹配 `release/1.2.0`
- 输入 `hotfix` 可以匹配 `hotfix/login-error`

## 数据设计

## 仓库本地统计

建议为仓库维护单独的本地统计记录，字段包括：

- `gitlabBaseUrl`
- `projectId`
- `projectName`
- `lastUsedAt`
- `useCount`

用途：

- 支撑仓库个性化排序
- 保持不同 GitLab 地址之间的数据隔离

## 分支数据

分支不做本地使用统计。

需要确保分支模型中保留以下信息：

- `name`
- `commitId`
- 最新提交时间字段

用途：

- 按最新提交时间排序
- 在列表中展示活跃度信息

## 组件与结构调整

建议拆分为更明确的显示单元：

- 新的仓库搜索选择组件
- 新的分支搜索选择组件
- 仓库排序与过滤辅助函数
- 分支排序与过滤辅助函数
- 仓库使用统计存储模块

保留现有：

- GitLab 请求层
- 当前标签页匹配逻辑
- 结果汇总区

## 文案建议

为了避免误导，仓库排序相关文案建议使用：

- `常用仓库`
- `最近使用`
- `按使用频率优先`

不建议使用：

- `提交最多`

原因：

- 当前仓库排序依据是插件本地使用记录
- 不是 GitLab 中真实的提交次数

分支排序文案可以强调：

- `按最近提交时间排序`
- `最近活跃分支优先`

## 测试策略

应覆盖以下场景：

- 仓库名称与路径搜索
- 仓库按 `useCount` 与 `lastUsedAt` 排序
- 当前标签页匹配仓库优先
- 分支按最新提交时间倒序排序
- 分支搜索过滤
- 仓库与分支空状态显示
- 选中仓库后更新本地统计
- 选中分支后更新结果汇总

## 风险

### 列表替换原生下拉后的交互复杂度上升

自定义列表比原生 `select` 更灵活，但需要自行处理高亮、空状态和点击行为。

缓解方式：

- 第一版先聚焦鼠标点击和基础搜索
- 键盘增强交互可作为下一阶段

### 本地排序可能让用户误以为是 GitLab 真实热度

缓解方式：

- 使用“常用”“最近使用”等文案
- 避免使用“提交最多”这样的表述

### 分支时间字段可能需要在映射层补充

缓解方式：

- 在 GitLab 分支数据映射时明确保留最新提交时间
- 通过单元测试锁定排序行为

## 完成标准

满足以下条件即可认为完成：

- 仓库区域支持搜索框 + 结果列表交互
- 仓库列表按本地使用次数与最近使用时间排序
- 分支区域支持搜索框 + 结果列表交互
- 分支列表按最新提交时间倒序排列
- 发布场景下能更快定位目标分支
- README 或相关测试能反映新的交互方式
