# 发布表单一键填入设计

## 目标

为当前 Chrome 插件增加“自动化填入相关信息”的能力。

用户在侧边栏中选好仓库和分支后，点击一个按钮，即可将以下信息一键写入当前打开的发布平台页面表单：

- git 仓库地址，使用带 `.git` 后缀的 clone 地址
- git 分支
- 最新提交 hash

同时，表单中的“仓库类型”下拉框需要先切换为 `git`，再执行后续填入。

## 背景

当前插件已经具备以下能力：

- 连接 GitLab
- 选择仓库
- 选择分支
- 展示仓库链接、分支和最新提交 hash
- 将这些值分别复制到剪贴板

但在实际提测或发布场景中，用户仍需要手动切回发布平台页面，逐项粘贴到弹窗表单中。这个流程重复、容易出错，也会打断当前操作节奏。

本次设计聚焦于把“复制”升级为“直接写入页面表单”，优先支持用户提供截图中的这一个发布平台页面。

## 范围

本次包含：

- 在侧边栏结果区新增“一键填入”按钮
- 点击按钮后向当前活动标签页注入脚本
- 先选择“仓库类型”为 `git`
- 再填入 git 仓库地址、git 分支、svn/git 修订号
- 为填入后的页面控件触发常见表单事件
- 为成功和失败结果提供侧边栏提示
- 补充注入逻辑与侧边栏交互测试

本次不包含：

- 支持多个不同发布平台页面
- 自动识别未打开的弹窗并主动打开
- 页面内悬浮按钮或常驻内容脚本 UI
- 用户自定义字段映射
- 自动提交表单

## 用户目标

用户希望：

- 在侧边栏选好仓库和分支后，一次点击完成页面填表
- 不需要手动复制和切换粘贴
- 如果页面结构不符合预期，能得到清晰错误提示

## 推荐方案

采用“侧边栏按钮触发 + 当前页面按固定规则注入”的方案。

原因：

- 当前插件的仓库、分支和 hash 已经在侧边栏中就绪，可以直接复用
- 只对当前活动标签页生效，操作范围明确，风险较小
- 不需要在页面中增加新的悬浮 UI，能保持页面侵入性最低
- 便于后续逐步扩展为更多页面模板

## 交互设计

## 侧边栏

在结果汇总区域保留现有复制按钮，并新增一个独立的“一键填入”按钮。

按钮启用条件：

- 当前已选中仓库
- 当前已选中分支
- 当前 hash 已加载完成

点击后：

1. 读取当前活动标签页
2. 通过 `chrome.scripting.executeScript` 向页面注入填表函数
3. 执行“先选仓库类型，再填字段”的固定流程
4. 将结果回传给侧边栏

成功提示建议文案：

- `已填入 git 链接、分支和 hash`

失败提示示例：

- `自动填入失败：未找到仓库类型下拉框`
- `自动填入失败：未找到 git 分支输入框`
- `自动填入失败：当前页面不支持注入`

## 页面填入流程

注入脚本执行顺序固定：

1. 定位“仓库类型”控件
2. 将其切换为 `git`
3. 定位“svn/git代码路径”输入框并填入带 `.git` 的 clone 地址
4. 定位“git分支”输入框并填入当前选中分支
5. 定位“svn/git修订号”输入框并填入当前 hash
6. 对每个写入动作触发 `input`、`change`，必要时补发 `blur`

这样可以尽量兼容常见前端框架对受控表单的监听。

## 选择器策略

由于当前目标页面是用户提供截图中的固定表单，本次不做抽象化配置，使用“标签文本驱动”的定向匹配策略。

匹配思路：

- 优先通过字段附近的中文标签文本定位目标表单项
- 在标签容器内查找对应的 `input` 或 `select`
- 对于下拉框，优先查找原生 `select`；若页面使用定制下拉，则优先尝试点击并匹配 `git` 选项

首批支持的字段标签：

- `仓库类型`
- `svn/git代码路径`
- `git分支`
- `svn/git修订号`

说明：

- 这里不依赖随机 class 名，优先依赖页面表单可见文案
- 如果页面改版导致标签文字变化，本功能可能失效，但错误会在侧边栏中明确暴露

## 模块设计

建议拆分为三个关注点清晰的小模块。

### 1. 侧边栏动作编排

职责：

- 控制“一键填入”按钮的可用状态
- 读取当前活动页标签信息
- 调用脚本注入
- 接收填入结果并转换为用户提示

建议位置：

- `src/sidepanel/App.tsx`
- `src/sidepanel/components/ResultSummary.tsx`

### 2. 页面自动填入工具

职责：

- 在目标页面中定位字段
- 选择 `git`
- 写入仓库地址、分支和 hash
- 触发表单事件
- 返回结构化结果

建议位置：

- `src/lib/autofill/releaseForm.ts`

### 3. 注入桥接

职责：

- 取得当前活动 tab
- 调用 `chrome.scripting.executeScript`
- 将侧边栏数据作为参数传入页面函数

建议位置：

- `src/lib/tabs/fillReleaseForm.ts`

## 数据结构

建议增加独立的填入请求与结果类型。

```ts
export type ReleaseFormAutofillPayload = {
  repositoryUrl: string;
  branch: string;
  commitHash: string;
};

export type ReleaseFormAutofillResult =
  | { ok: true }
  | { ok: false; reason: string };
```

这样可以让页面逻辑和侧边栏逻辑之间的接口更稳定，也便于测试。

## 错误处理

自动填入过程中的错误分为三类：

### 页面环境错误

例如：

- 当前没有活动标签页
- 当前标签页为浏览器内部页面，无法注入

处理方式：

- 在桥接层统一转换成用户可读错误

### 页面结构错误

例如：

- 未找到“仓库类型”下拉
- 未找到“git分支”输入框

处理方式：

- 页面函数返回明确的 `reason`
- 侧边栏直接展示该原因

### 数据状态错误

例如：

- 仓库地址为空
- 分支为空
- hash 尚未加载

处理方式：

- 在侧边栏按钮层直接禁用，不进入注入流程

## 测试策略

测试分两层。

## 纯函数 / DOM 工具测试

针对页面自动填入模块补充单元测试，覆盖：

- 先选择 `git` 再填字段
- git 地址按带 `.git` 的值写入
- 找不到必填字段时返回明确失败原因
- 写入后触发 `input` 与 `change` 事件

## 侧边栏集成测试

针对侧边栏交互补充测试，覆盖：

- 数据未就绪时“一键填入”按钮禁用
- 数据就绪后点击按钮触发注入
- 注入成功时展示成功提示
- 注入失败时展示失败提示

## 风险与兼容性

主要风险：

- 发布平台使用非原生下拉，导致“仓库类型”切换需要额外兼容
- 页面标签文案或结构调整后，字段定位可能失效
- 某些受控表单仅靠设置 `value` 不足以同步状态，需要额外触发事件

缓解方式：

- DOM 工具优先通过标签文本匹配，不依赖样式 class
- 失败原因做成明确文案，方便后续快速修正
- 将页面填表逻辑集中在单独模块中，后续改选择器时影响面最小
